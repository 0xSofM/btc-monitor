# BTC监控系统数据流分析报告

## 一、数据更新、历史存储及前端同步逻辑梳理

### 1.1 数据获取流程

#### 后端自动更新服务 (`auto_update_service.py`)
```
1. 定时触发（默认10分钟）→ 2. API请求5个指标 → 3. 数据处理与合并 → 4. 保存到JSON文件
```

**核心逻辑：**
- 获取5个API端点数据：`/v1/mvrv-zscore`, `/v1/lth-mvrv`, `/v1/puell-multiple`, `/v1/nupl`, `/v1/btc-price`
- 合并所有日期，计算200周均线（需要1400天价格数据）
- 向前填充缺失值（`_get_latest_value`方法）
- 保存到两个位置：`app/public/btc_indicators_history.json` 和根目录备份

#### 前端数据服务 (`dataService.ts`)
```
1. 优先级：静态文件 → 实时API → 历史回退 → 本地存储
2. 缓存机制：内存缓存1分钟，本地存储持久化
3. 自动刷新：5分钟间隔
```

### 1.2 数据存储架构

#### 文件存储
- `btc_indicators_history.json`: 完整历史数据（6286条记录）
- `btc_indicators_latest.json`: 最新数据摘要
- `app/public/`: 前端可访问的静态文件

#### 前端缓存
- 内存缓存：`cache.data` 和 `cache.history`
- 本地存储：`localStorage` 中的 `btc_indicators_history` 和 `btc_indicators_latest`

### 1.3 前端展示逻辑

#### 数据源优先级 (`App.tsx`)
```typescript
1. 自动模式：优先静态文件 → API → 历史回退
2. 手动模式：直接API → 静态文件 → 历史回退
```

#### 五个核心指标展示
1. **BTC Price / 200W-MA**: 价格与200周均线比值
2. **MVRV Z-Score**: 市值相对实现价值偏离
3. **LTH-MVRV**: 长期持有者盈亏状态
4. **Puell Multiple**: 矿工收入相对历史均值
5. **NUPL**: 全网未实现盈利/亏损状态

## 二、发现的问题与分析

### 2.1 API限流问题
**现象**: API返回429错误，触发限流保护
**影响**: 无法获取实时数据进行一致性校验
**现状**: 自动更新服务正常（已获取最新数据），但直接API请求被限流

### 2.2 数据日期不一致问题
**现象**: 不同指标的数据日期存在差异
**原因**: 
- 各指标API更新频率不同
- `indicatorDates` 字段记录实际数据日期
- 向前填充机制导致展示日期与实际数据日期不符

### 2.3 前端数据同步问题
**潜在风险**:
- 本地存储与展示数据可能不一致
- 缓存失效时数据源切换逻辑
- 多数据源版本冲突

2. **日期一致性**: 在前端更清晰地标示各指标的实际数据日期
